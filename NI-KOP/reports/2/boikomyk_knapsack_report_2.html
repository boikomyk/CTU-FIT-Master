1.2 Dynamic programming¶

Hlavn mylenka Dynamic Programming v ppad problm batohu je pouit tabulky kam se ukldaj een vech podproblmu (vechny mon hodnoty cen/vah, v zvislosti na zvolenm typu dekompozici, kter mohou vzninkout pi pidn vci do batohu). Pokud v prbhu een znovu narme na ji eeny podproblm, tak sta vzt een v tabulce, ani byste podproblm museli znovu eit.

Je teba poznamenat, e nalezen een je vdy optimln, nebo algoritmus vytvo tabulku pro cel strom vech monch vsledk. Stejn je nutn ct, e algoritmus je pseudopolynomiln, jeliko doba bhu je O(NW) nebo O(N*sum_of_all_items_prices) pro neomezen problm batohu s N polokami a batohem velikosti W (W vak nen polynomiln vi dlce vstupu). Co u klidn umouje Nm odpovdt na jeden z kladench dotaz, e sloitost algoritmu zvis na parametru, kter nem nic spolenho s velikost.

1.2.1 Decomposition by weight¶

V ppad dokompozice podle vhy tabulka m nalsedujci parametry:

  • poet dku je stejn jako celkov poet vci dan pro danou instanci problm

  • poet sloupc je rovn kapacit/nosnosti batohu (maximlani povolena vha)

Jinak eeno, to znamen, e kad dek reprezentuje een problm pro poet pedmt, kter se rovn indexu dku. Stejn tak sloupce reprezentuj vechny mon vsledky vah (pro vechny kombinace vech sprvnch een). Co vede na to, e buka[i,j] reprezentuje cenu takov kombinace. Finln vsledek je uloeny v poslednm dku (maximln hodnota v poslednm dku).

Tabulka se stav nsledovn: (Zkrcen kd..)

for item_index in range(items_indexes_rows_cnt + 1):
    for capacity_index in range(capacity_columns_cnt + 1):
        ...
        #  if weight of current item is lower or equal to the remaining knapsack's capacity then
        if current_item.weight <= capacity_index:
            # return the maximum of two cases:
            # - current item included or
            # - not included
            memory_table[item_index][capacity_index] = max(
                current_item.price + memory_table[item_index - 1][capacity_index - current_item.weight],
                memory_table[item_index - 1][capacity_index]
            )
        # otherwise current proceeding item cannot be included
        else:
            memory_table[item_index][capacity_index] = memory_table[item_index - 1][capacity_index]

1.2.2 Decomposition by price¶

V podstat pstup k een je skoro ten samy, avak je nutn poznamenat dal dleit rozdly. Tabulka m nalsedujci parametry:

  • poet dku je stejn jako celkov poet vci dan pro danou instanci problm

  • poet sloupc je rovn sum cen vech vci dan instance problmu.

Finln vsledek je zase uloeny v poslednm dku (maximln hodnota, avak je nutn aby byla splnna podmnka: hodnota_buky <= knapsack_capacity)

Tabulka se stav zase nsledovn: (Zkrcen kd..):

for knapsack_current_price in range(2, items_total_sum + 1):
    for item_index in range(1, items_total_count + 1):
        ...

        index = knapsack_current_price - current_item.price
        if index < 0:
            # if index is negative, the weight remains the same
            memory_table[knapsack_current_price][item_index] = previous_weight
        else:
            # otherwise
            # resolve weight to add (zero or weight of current proceeding item)
            weight_to_add = current_item.weight if memory_table[index][item_index - 1] != infinity else 0
            # W(c, i) = min(W(c, i - 1), W(c, i - 1 ) + w_c) for any c > 0
            memory_table[knapsack_current_price][item_index] = min(previous_weight, memory_table[index][item_index - 1] + weight_to_add)

1.3 Greedy¶

Hlavni myslenka Greedy pristupu k reseni je super jednoducha: seadit vechny veci sestupn podle jejich pomru: $$ pomr = \frac{cena}{vaha} $$

a pidvat veci postupn, dokud nen batoh pln.

Je nejrychlej metodou pro een batohu, jej sloitost zvis na sloitosti pouitho azen (zpravidla je O(nlogn)).

Ale nen zand 100% garance, e nalezen een bude optimln, nebo teba pouze jedna vc s maximln cenou a zrove vahou me bt eenm, ale kvli nzkmu pomru nejsp bude ignorovna algoritmem. (Tento ppad e modifikace Redux, kter bude popsna troku dl)

U tohoto algoritmu je nutn zavst dal pojem: Relativn chyba, kter se vypot nsledovn: $$ chyba = \frac{|cena_{een} - cena_{reference}|}{max(cena_{een}, cena_{reference})} $$

1.4 Redux¶

Redux je vylepenm(/modifikace) pedchoz heuristiky (vlastn e ten ppad s jednou vci s maximln cenou a vahou).

Nejdv pro danou instanci nalezneme een popsanou Greedy heuristikou. Pak vytvome dal batoh s jednou nejrdazsi vci a porovnme ob een. Lep z tchto een (s vt cenou nebo stejnou cenou ale men vahou) je pak eenm finlnm.

1.5 FPTAS¶

Hlavn mylenka algoritmu, v kominaci s Dynamic programming s dekompozici podle ceny, je snen celkovho prostoru pro prohledvn za pedpokladu toho, e dovolujeme algoritmu dlat chyby.

Zavdme maximln relativn chybu pro algoritmus a zmenujeme sumu cen vech vci danho problm, tm vlastn zmenujeme velikost tabulky (kde, je nutn pipomenout, poet sloupc je rovn sum cen vech vci). Co vede na to, e suboptimln een bude nalezeno vrazn rychleji.

Takhle se modifikuj vci, ped tm jak eit instanci pomoci Dynamic programmingu: (Zkrcen kd..):

# resolve knapsack item with greatest price
max_price = max(item.price for item in knapsack_modified.items_list)
# calculate the denominator: K = (eps * C_M) / items_cnt
K = (error * max_price) / len(knapsack_modified.items_list)
# than build a modified items version: decrease prices
for modified_item in knapsack_modified.items_list:
    modified_item.price = int(modified_item.price / K)

# than modify knapsack object
knapsack_modified.min_req_price = int(knapsack_modified.min_req_price / K)
# solve modified knapsack instance with dynamic programming with decomposition by price
configurations_cnt, solution = Resolver.dynamic_programming_decomposition_by_price(
    knapsack=knapsack_modified
)

def read_and_store_csv_input_to_df(path_to_csv: str, algorithm: str, instance_size: int, accuracy_value = None):
    # read csv input

def display_basic_dataframe_analyse(
    df_data: pd.DataFrame,
    additional_values_to_analyze: List[str] = [],
    additional_indexes: List[str] = []
):
    # default values to analyze: STEPS and TIME    
    values = ['STEPS', 'TIME[ms]', 'ERROR'] + additional_values_to_analyze
    # default indexes: ALGORITHM and INSTANCE_SIZE
    indexes = ['ALGORITHM', 'INSTANCE_SIZE'] + additional_indexes
    # default funcs to execute on values: min, mean, max and standard deviation
    funcs = [np.min, np.mean, np.max, np.std]
    
    statistics_for_algorithms = ''
    for algorithm_key in df_data['ALGORITHM'].unique():
        statistics_for_algorithms += f', {ALGORITHMS[algorithm_key]}' if len(statistics_for_algorithms) else ALGORITHMS[algorithm_key]

    print(f"="*100)
    print(f"{statistics_for_algorithms} algorithm: ")

    display(pd.pivot_table(
        data=df_data,
        values=values,
        index=indexes,
        aggfunc={value: funcs for value in values}
    ))
    print(f"="*100)

Vlastn Dynamic Programming je algoritmus s relativn dobrou sloitost, ale vysokou pamovou nronost. Maximln sloitost algoritmu je zvisl na souinu poet vci*nosnost batohu(/celkov suma vci).

Jak ji bylo zmnno dve, algoritmus je optimln a vdy najde nejlep optimln een, nebo stavme tabulku celho prostoru vech monch een, plus je vidt z vypsan statistiky v, e maximln chyba pro oba dva typy dekompozice je nulov.

Pro vtinu instanc problm plat, e velikost(/nosnost) batohu je striktn men ne suma cen vech vci, proto tabulku, kterou stavme pi dekompozici podle vhy, mla by bt men velikosti a stejn tak, as kter potebujeme pro stavbu takov tabulky bude mnohem men.

6. Závěr¶

Bhem tto prci bylo implementovno spoustu novch algoritmu a prozkoumno novch pstupu k een problm bathou. Taky jsem seznmil s pojemem relativn chyby u heruistickych algoritmu. Stejn tak jsem se nauil takovm optimalizanm trikm, jak Redux nadstavba nad jednoduchm Greedy algoritmem, kter docela siln znizuje relativn chybu, a FPTAS nadstavba nad Dynamic Programming, kter docela zrychluje pvodn algorimus, ale s tm pin relativn chybu.

Dynamic Programming s dekompozici podle vhy a Branch & Bounds se ukzaly jako nejrychlej algoritmy, kter navc vdy dokou zskat optimln nejlep een.
